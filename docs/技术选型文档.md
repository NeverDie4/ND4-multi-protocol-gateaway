# 技术选型文档

> 当前文档以项目现有实现为准：后端为 Go + Gin + GORM，前端保留并采用 `mount-hub/` 目录下的 Next.js 实现。根目录下 `web/` 暂不作为当前前端实现依据。

---

## 1. 总体技术路线

本项目采用“Go 后端 + Next.js 前端”的前后端分离开发、后端统一承载协议服务的架构。

| 层级 | 当前选型 | 说明 |
|---|---|---|
| 后端语言 | Go | 负责 REST API、存储驱动、协议服务、任务调度、数据库访问 |
| 后端框架 | Gin | 提供 HTTP API、下载代理、静态资源服务 |
| CLI 框架 | Cobra | 提供 `server`、`admin`、`storage`、`mcp` 等命令 |
| ORM | GORM | 通过 AutoMigrate 初始化数据库表结构 |
| 前端框架 | Next.js 16 | 当前前端实现位于 `mount-hub/`，使用 App Router |
| 前端语言 | TypeScript | 页面、组件、接口类型统一使用 TypeScript |
| UI 体系 | shadcn/ui + Radix UI + Tailwind CSS | 当前 `mount-hub/` 已按该体系组织组件 |
| 包管理器 | pnpm | `mount-hub/` 使用 `pnpm-lock.yaml` 管理依赖 |

## 2. 后端技术选型

### 2.1 运行时与框架

| 类别 | 选型 | 说明 |
|---|---|---|
| 语言 | Go | 适合高并发 IO、流式传输和单文件部署 |
| HTTP 框架 | Gin | API 路由、中间件、上传下载、静态资源分发 |
| CLI | Cobra | 管理服务启动、管理员命令、存储命令等 |
| ORM | GORM | 负责模型映射、数据库迁移和基础 CRUD |
| 日志 | logrus | 统一服务日志输出 |
| 配置 | JSON 配置 + 环境变量 | 支持本地配置文件和运行环境覆盖 |

### 2.2 数据库

项目通过 GORM AutoMigrate 预设表结构，而不是维护独立 SQL 初始化脚本。

当前核心表包括：

| 表 | Model | 用途 |
|---|---|---|
| `x_storages` | Storage | 存储挂载配置 |
| `x_users` | User | 用户、密码、角色、2FA、SSO 信息 |
| `x_roles` | Role | 角色和路径级权限 |
| `x_metas` | Meta | 路径元数据、密码、隐藏规则、README、公告 |
| `x_setting_items` | SettingItem | 系统配置项 |
| `x_search_nodes` | SearchNode | 搜索索引节点 |
| `x_task_items` | TaskItem | 异步任务记录 |
| `x_ssh_public_keys` | SSHPublicKey | SFTP 公钥认证 |
| `x_labels` | Label | 文件标签 |
| `x_label_file_bindings` | LabelFileBinding | 标签与文件绑定 |
| `x_obj_files` | ObjFile | 文件对象缓存 |
| `x_sessions` | Session | 用户设备会话 |
| `x_shares` | Share | 文件分享 |

默认表名前缀为 `x_`，可通过配置调整。

### 2.3 协议与文件能力

| 能力 | 当前实现 |
|---|---|
| HTTP REST API | Gin 路由，集中在 `server/router.go` 和 `server/handles/` |
| WebDAV | `server/webdav/` 与 `/dav` 路由 |
| FTP | `server/ftp.go` 与 `server/ftp/` |
| SFTP | `server/sftp.go` 与 `server/sftp/` |
| S3 兼容服务 | `server/s3.go` 与 `server/s3/` |
| MCP | `server/mcp/` |
| 文件上传 | PUT 流式上传、multipart 上传 |
| 文件下载 | 直链跳转、后端代理、Range、限速 |
| 异步任务 | 上传、复制、离线下载、解压等任务管理 |

## 3. 前端技术选型

### 3.1 当前前端实现位置

当前正式保留的前端目录为：

```text
mount-hub/
```

根目录下 `web/` 是另一套 Vite/React 结构，当前先忽略，不作为技术选型和验收依据。

### 3.2 框架与构建

| 类别 | 当前选型 | 实际依据 |
|---|---|---|
| 前端框架 | Next.js | `mount-hub/package.json` 中依赖 `next` |
| 路由 | Next.js App Router | 页面位于 `mount-hub/src/app/` |
| 语言 | TypeScript | `tsconfig.json`、`.tsx` 页面与组件 |
| React | React 19 | `react`、`react-dom` 依赖为 19 系 |
| 包管理 | pnpm | 存在 `pnpm-lock.yaml` |
| 构建命令 | `next build` | `package.json` 中定义 |
| 开发命令 | `next dev` | `package.json` 中定义 |

当前前端不再按 Vite SPA 方案描述，也不再要求 `TanStack Router` 作为路由方案。

### 3.3 目录结构

`mount-hub/` 当前主要结构如下：

```text
mount-hub/
├─ package.json
├─ pnpm-lock.yaml
├─ next.config.mjs
├─ tsconfig.json
├─ postcss.config.mjs
├─ components.json
├─ public/
├─ src/
│  ├─ app/
│  │  ├─ layout.tsx
│  │  ├─ page.tsx
│  │  ├─ login/
│  │  ├─ files/
│  │  ├─ admin/
│  │  └─ workspace/
│  ├─ components/
│  │  ├─ admin/
│  │  ├─ common/
│  │  ├─ file/
│  │  └─ ui/
│  ├─ hooks/
│  ├─ lib/
│  └─ types/
```

说明：

- `src/app/` 是 Next.js App Router 的主要页面入口。
- `src/components/ui/` 是 shadcn/ui 风格组件。
- `src/lib/api.ts` 当前仍为 mock API 封装，后续需要替换为真实后端 API 调用。
- 当前同时存在顶层 `app/` 和 `src/app/`，后续建议统一保留 `src/app/`，避免入口混淆。

### 3.4 UI 与样式

| 类别 | 当前选型 | 说明 |
|---|---|---|
| UI 组件 | shadcn/ui | 组件代码复制到本地，可直接修改 |
| 基础组件 | Radix UI | Dialog、Dropdown、Tabs、Select、Toast 等 |
| CSS | Tailwind CSS 4 | 原子化 CSS 与 CSS Variables |
| 图标 | lucide-react | 管理端、文件操作、导航等图标 |
| 主题 | next-themes + CSS Variables | 支持明暗主题扩展 |
| 通知 | sonner | Toast 通知 |

### 3.5 状态管理与数据请求

当前实现与原文档相比有明显变化：

| 类别 | 原文档设想 | 当前实际 |
|---|---|---|
| API 客户端 | Axios | 当前 `src/lib/api.ts` 使用 mock 数据 |
| 服务端状态 | TanStack Query | 当前未接入 |
| 客户端状态 | Zustand | 当前认证状态使用 React Context |
| 表单 | React Hook Form + Zod | 依赖已存在，可继续采用 |

当前建议：

- 短期：保留 React Context，先把 mock API 替换为真实 API。
- 中期：如果文件列表、任务、用户管理等接口需要缓存、轮询和重试，再引入 TanStack Query。
- 是否引入 Zustand 可后置，不作为当前硬性要求。

### 3.6 路由方案

当前路由采用 Next.js App Router，而不是 TanStack Router。

| 页面 | 当前路径 | 说明 |
|---|---|---|
| 首页跳转 | `/` | 根据登录状态跳转 |
| 登录页 | `/login` | 用户登录 |
| 文件页 | `/files` | 文件浏览与操作 |
| 管理页 | `/admin` | 存储、用户、协议、限速等管理 |
| 工作台 | `/workspace` | 工作台入口 |

Next.js 路由由目录结构决定，不再维护独立 `routes/` 目录。

### 3.7 与 Go 后端的集成方式

当前需要重点处理 Next.js 与 Go 后端静态资源发布方式的差异。

原 Vite 方案可以直接构建到：

```text
public/dist/
```

但 Next.js 默认构建产物为：

```text
mount-hub/.next/
```

因此后续有两种可选集成路线：

| 路线 | 说明 | 适用场景 |
|---|---|---|
| Next.js 独立部署 | `next start` 单独运行，Go 后端提供 API 和协议服务 | 开发快，保留 SSR/App Router 能力 |
| Next.js 静态导出 | 配置 `output: 'export'`，导出静态文件后接入 Go 的 `public/dist` | 更接近原 AList 单二进制部署模式 |

如果项目目标是“Go 单服务内嵌前端”，建议后续评估 Next.js 静态导出，并处理动态路由、图片优化、API base path 等限制。

### 3.8 当前不再作为硬性要求的前端库

以下库在旧文档中出现，但 `mount-hub/` 当前未实际采用，因此不再作为当前阶段硬性选型：

| 原计划库 | 当前处理 |
|---|---|
| TanStack Router | 已由 Next.js App Router 替代 |
| TanStack Query | 暂未接入，可按需求后续引入 |
| Zustand | 暂未接入，可按复杂度后续引入 |
| Axios | 暂未接入，后续真实 API 可选择 `fetch` 或 Axios |
| motion | 暂未接入，不作为首期硬性要求 |
| react-virtuoso | 暂未接入，文件量较大时再引入 |
| @dnd-kit/core / sortable | 暂未接入，拖拽需求明确后再引入 |
| vidstack / lightbox / monaco / react-markdown | 暂未接入，媒体预览增强阶段再评估 |
| @antfu/eslint-config / husky / lint-staged | 暂未接入，可作为工程规范增强项 |

## 4. 前后端通信

### 4.1 API 约定

Go 后端 API 统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

需要注意：后端多数业务错误也会返回 HTTP 200，真实业务状态通过 JSON 中的 `code` 判断。

### 4.2 主要接口

| 场景 | 后端接口 |
|---|---|
| 登录 | `POST /api/auth/login/hash` |
| 当前用户 | `GET /api/me` |
| 文件列表 | `POST /api/fs/list` |
| 文件详情 | `POST /api/fs/get` |
| 创建目录 | `POST /api/fs/mkdir` |
| 重命名 | `POST /api/fs/rename` |
| 移动 | `POST /api/fs/move` |
| 复制 | `POST /api/fs/copy` |
| 删除 | `POST /api/fs/remove` |
| 上传 | `PUT /api/fs/put` 或 `PUT /api/fs/form` |
| 下载 | `GET /d/*path` 或 `GET /p/*path` |
| 存储管理 | `/api/admin/storage/*` |
| 用户管理 | `/api/admin/user/*` |
| 角色管理 | `/api/admin/role/*` |
| 系统设置 | `/api/admin/setting/*` |
| 分享 | `/api/share/*` 与 `/api/public/share/*` |

### 4.3 认证方式

| 方式 | 用途 |
|---|---|
| JWT Token | Web 前端登录后访问 API |
| Admin Raw Token | 管理脚本或超级权限场景 |
| Basic Auth | WebDAV、FTP、SFTP 等协议接入 |
| 2FA | 用户登录二次验证 |
| SSO / LDAP / WebAuthn | 企业登录扩展 |

前端真实 API 接入时，应在请求层统一处理：

- `Authorization` 请求头。
- `Client-Id` 设备标识。
- 后端 JSON `code` 非 200 的错误提示。
- 登录失效、设备会话失效、权限不足等状态。

## 5. 开发与构建流程

### 5.1 本地开发

后端：

```bash
go run . server --dev
```

前端：

```bash
cd mount-hub
pnpm install
pnpm dev
```

默认情况下，Next.js 开发服务运行在：

```text
http://localhost:3000
```

Go 后端默认服务端口按项目配置，一般为：

```text
http://localhost:5244
```

### 5.2 生产构建

当前 `mount-hub/` 的生产构建命令为：

```bash
cd mount-hub
pnpm build
```

该命令会生成 Next.js 构建产物 `.next/`，并不等同于 Go 后端当前期望的 `public/dist/`。

如果要实现 Go 后端内嵌前端，需要后续补充构建策略，例如：

- 配置 Next.js 静态导出。
- 将导出结果复制到 `public/dist/`。
- 校验 `server/static` 对 `index.html`、`assets/`、`images/`、`static/` 等目录的要求。

## 6. 当前一致性结论

当前项目应以 `mount-hub/` 的技术路线为准，前端选型应修正为：

```text
Next.js 16 + App Router + React 19 + TypeScript + Tailwind CSS 4 + shadcn/ui + Radix UI
```

旧文档中的 Vite、TanStack Router、TanStack Query、Zustand、Axios、motion、react-virtuoso 等内容，不再作为当前已经实现的事实描述。其中 TanStack Query、Axios、Zustand 等可以作为后续增强选项，但不应写成当前已采用技术。

当前仍需补齐的前端工程事项：

1. 将 `src/lib/api.ts` 从 mock 数据切换到真实 Go API。
2. 明确 Next.js 与 Go 后端的部署关系。
3. 清理或统一顶层 `app/` 与 `src/app/` 的重复入口。
4. 决定是否需要静态导出到 `public/dist/`。
5. 提交前排除 `.next/`、`node_modules/` 等构建和依赖目录。
