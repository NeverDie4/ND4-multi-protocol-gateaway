| 掌握长连接、流式传输、高性能网络 IO | 文件访问支持流式下载、Range 分块、代理传输 |
| 协议代理与转换 | 用户通过统一入口访问，平台解析路径后转发到真实存储并返回流 |

|------|------|------|

| B3 | 网页门户：浏览、下载、上传、目录管理 | 前后端与文件操作全流程打通 |
| B4 | 网页端 HTTPS 加密传输，SSL/TLS 证书配置 | HTTPS 可访问，证书配置生效 |

## 2. 当前实现详析

### 2.1 总体架构

```
┌────────────────────────────────────────────────────────────────┐
│                         客户端                                  │
│   浏览器 (HTTPS)  │  Win资源管理器 (WebDAV)  │  FTP/SFTP 客户端  │
└───────┬──────────┬───────────┬──────────────┬──────────────────┘
        │          │           │              │
        ▼          ▼           ▼              ▼
   ┌──────── ┌───────── ┌────────── ┌──────────────┐
   │ HTTP/s  │  WebDAV  │    S3     │  FTP / SFTP   │
   │ :5244   │  /dav    │   :5246   │  :5221 / :5222│
   └────┬────┴────┬─────┴─────┬─────┴──────┬────────┘
        │         │           │            │
        ▼         ▼           ▼            ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Gin Router + Middleware                      │
│  Auth(JWT/Token/Bearer) → Sign → RateLimit → FsCheck → Handler  │
├─────────────────────────────────────────────────────────────────┤
│                      fs 层: 路径解析 → 存储映射                    │
│                      op 层: 缓存 → 排序 → 单飞(去重)               │
├─────────────────────────────────────────────────────────────────┤
│                   统一存储接口 (driver.Driver)                     │
│  ┌────────┬────────┬────────┬────────┬────────┬──────────┐      │
│  │ Local  │  FTP   │ WebDAV │  SFTP  │   S3   │ Azure    │ ...  │
│  │        │        │ 客户端 │        │        │ Blob     │      │
│  └────────┴────────┴────────┴────────┴────────┴──────────┘      │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 请求数据流（以文件下载为例）

```
1. 用户请求: GET /d/local/docs/report.pdf
2. Auth 中间件 → 解析 JWT → 注入 user 到 context
3. Sign 中间件 → 校验下载签名
4. RateLimit 中间件 → 包裹 ResponseWriter 为限速写入器
5. handler.Down() → fs.Link("/local/docs/report.pdf")
6. op.GetStorageAndActualPath(path)
   → storagesMap 最长路径匹配 → Local 驱动
   → 实际路径: "/docs/report.pdf"
7. op.Get() (缓存优先) → driver.Get() / driver.List()
8. driver.Link() → os.Open(fullPath) → MFile (io.ReadSeekCloser)
9. proxy.Proxy() → http.ServeContent(w, r, name, modTime, MFile)
10. 响应 → 支持 Range → 206 Partial Content → 限速流式写入
```

#### 3.7.2 权限控制

**相关代码**: `internal/model/role.go`, `server/common/role_perm.go`

权限模型: **Role → PermissionScope（路径 + 位掩码）→ User**

**17 种权限位**:

| 位 | 常量 | 含义 |
|----|------|------|
| 0 | `PermSeeHides` | 可见隐藏文件 |
| 1 | `PermAccessWithoutPassword` | 免密码访问 |
| 2 | `PermAddOfflineDownload` | 离线下载 |
| 3 | `PermWrite` | 写入/上传 |
| 4 | `PermRename` | 重命名 |
| 5 | `PermMove` | 移动 |
| 6 | `PermCopy` | 复制 |
| 7 | `PermRemove` | 删除 |
| 8 | `PermWebdavRead` | WebDAV 读 |
| 9 | `PermWebdavManage` | WebDAV 写 |
| 10 | `PermFTPAccess` | FTP/SFTP 登录 |
| 11 | `PermFTPManage` | FTP/SFTP 写 |
| 12 | `PermReadArchives` | 读取压缩包 |
| 13 | `PermDecompress` | 解压 |
| 14 | `PermPathLimit` | 路径限制 |
| 15 | `PermMCPAccess` | MCP 读 |
| 16 | `PermMCPManage` | MCP 写 |

**权限匹配逻辑**:

- 请求路径 `/a/b/c` → 遍历用户所有 Role 的 PermissionScope
- 匹配规则: `scope.Path` 是 `reqPath` 的上级路径时合并该 scope 的所有权限位
- 可见权限: 无 `PermSeeHides` 的用户，目录中的文件按 `meta.Hide` 正则过滤

|------|------|------|

| 读权限控制 | 完成 | `CanReadPathByRole()` 路径可达性检查 |

## 7. 限制说明

1. **网盘挂载**: 用户已明确排除，项目中原有的 80+ 网盘驱动（阿里云盘、OneDrive、百度网盘等）不在本次分析范围内。
2. **Go 编译环境**: 当前系统 PATH 中未检测到 `go` 命令，开发环境搭建需在验收前完成。

## 8. 总结

从代码层面看，当前项目已具备多协议文件挂载平台的完整后端能力，在协议类型和存储驱动数量上远超实验最低要求。真正影响交付完整性的不是核心代码能力，而是以下三类问题：

1. **交付缺口**: 前端 `public/dist` 缺失，网页门户无法直接验收
2. **验收缺口**: HTTPS 配置样例、WebDAV Win/Mac 客户端连接缺少可复现验证材料
3. **功能细化缺口**: 下载限速只有全局能力，"针对不同用户"的 QoS 限速需进一步增强

建议优先补齐 P0 项（前端资源、Go 环境），再依次处理 P1 项（用户级限速、HTTPS 文档、WebDAV 验收）。
